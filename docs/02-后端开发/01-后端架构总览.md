# 后端架构总览

## 一、架构分层

后端采用经典的三层架构模式，通过 FastAPI 的依赖注入实现各层解耦。

```
┌─────────────────────────────────────────┐
│           API 层 (Routes)               │
│  路由分发、参数校验、响应包装        │
└──────────────────────┬──────────────────┘
                       │
┌──────────────────────▼──────────────────┐
│         业务逻辑层 (Implicit)           │
│  业务规则、数据处理、事务管理        │
│  (当前直接在路由中实现，可抽离 Service) │
└──────────────────────┬──────────────────┘
                       │
┌──────────────────────▼──────────────────┐
│        数据访问层 (Models + ORM)        │
│  SQLAlchemy ORM、数据模型定义        │
└──────────────────────┬──────────────────┘
                       │
┌──────────────────────▼──────────────────┐
│           数据存储层                    │
│  MySQL / Redis / MinIO               │
└─────────────────────────────────────────┘
```

### 1.1 各层职责

| 层级 | 对应目录 | 职责 |
|------|---------|------|
| API 层 | `app/api/v1/` | 定义路由、请求参数校验（Pydantic）、响应格式包装 |
| 业务逻辑层 | 路由函数内（可抽离 service） | 业务规则、流程编排、事务控制 |
| 数据访问层 | `app/models/` | SQLAlchemy ORM 模型定义、数据库操作 |
| 基础设施层 | `app/core/` | 配置、数据库连接、安全工具等通用能力 |

## 二、应用入口

### 2.1 main.py 核心结构

[main.py](file:///g:/资料/code/factory-pms/backend/app/main.py) 是 FastAPI 应用的入口文件，主要完成：

1. **创建 FastAPI 实例**
   - 配置应用名称、描述、版本
   - 设置 OpenAPI 文档路径

2. **配置中间件**
   - CORS 跨域中间件

3. **注册路由**
   - 按模块注册 v1 版本的所有 API 路由

4. **健康检查**
   - 根路径 `/` 返回欢迎信息
   - `/health` 健康检查端点

### 2.2 启动流程

```
启动 Uvicorn
    ↓
加载 main.py → 创建 FastAPI 实例
    ↓
加载 settings 配置（环境变量/.env）
    ↓
初始化数据库连接池 (SQLAlchemy Engine)
    ↓
注册所有路由
    ↓
启动 HTTP 服务，监听端口
```

## 四、请求处理流程

### 4.1 典型请求生命周期

```
HTTP Request
    ↓
FastAPI 接收
    ↓
CORS 中间件处理
    ↓
路由匹配
    ↓
Pydantic 参数校验 (Query/Path/Body)
    ↓
依赖注入执行
  ├── get_db() → 获取数据库会话
  └── get_current_user() → JWT 认证 (可选)
    ↓
路由函数执行
  ├── 业务逻辑处理
  ├── 数据库操作 (通过 ORM)
  └── 数据组装
    ↓
Pydantic 响应序列化
    ↓
HTTP Response
```

### 4.2 认证请求示例

以 `GET /api/v1/projects/list` 为例：

1. 请求携带 `Authorization: Bearer <token>` 头
2. `oauth2_scheme` 提取 token
3. `get_current_user` 解码 JWT，查询用户
4. 验证用户状态（is_active）
5. 路由函数获取 `current_user` 对象
6. 执行业务逻辑

## 五、响应格式规范

### 5.1 统一响应结构

所有 API 响应使用统一的包装格式：

```python
{
    "code": 200,           # 状态码：200成功，其他为错误码
    "message": "success",  # 提示信息
    "data": {}             # 业务数据（可能是对象、数组、分页等）
}
```

对应 Pydantic 模型：[ResponseBase](file:///g:/资料/code/factory-pms/backend/app/schemas/common.py#L7-L13)

### 5.2 分页响应结构

```python
{
    "items": [],           # 数据列表
    "total": 100,          # 总条数
    "page": 1,             # 当前页
    "page_size": 20,       # 每页大小
    "total_pages": 5       # 总页数
}
```

对应 Pydantic 模型：[PaginationResponse](file:///g:/资料/code/factory-pms/backend/app/schemas/common.py#L42-L47)

## 六、错误处理

### 6.1 HTTP 状态码

| 状态码 | 场景 |
|--------|------|
| 200 | 请求成功 |
| 400 | 参数错误 / 业务校验失败 |
| 401 | 未认证 / token 无效 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 422 | 请求参数校验失败（FastAPI 自动） |
| 500 | 服务器内部错误 |

### 6.2 异常抛出方式

使用 FastAPI 的 `HTTPException`：

```python
from fastapi import HTTPException

# 资源不存在
raise HTTPException(status_code=404, detail="项目不存在")

# 权限不足
raise HTTPException(status_code=403, detail="权限不足")

# 参数错误
raise HTTPException(status_code=400, detail="用户名已存在")
```
